Building Apps

Installation Flow

How users install your app, provide credentials, and what states an installation goes through

An installation is the link between a user account and your app. Orceum tracks its state, stores credentials securely, and uses it to route action calls on behalf of the user.

​
The Install Endpoint

Installations are created programmatically via the Orceum SDK or directly by the platform when a user clicks “Install” in the UI. 
curl -X POST https://api.orceum.com/v1/apps/{app_id}/install \
  -H "Authorization: Bearer USER_ORCEUM_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{}'

​
Response

{
  "installation_id": "inst_abc123",
  "app_id": "app_a1b2c3d4",
  "status": "PENDING",
  "next_action": "NONE",
  "created_at": "2024-01-15T12:00:00Z"
}

​
next_action — What Happens After Install

The next_action in the response tells the Orceum frontend what to do next:
next_actionMeaningWhat Happens
NONEInstallation is completeApp is immediately ready to use
API_KEYAPI key requiredFrontend shows the API key input form
OAUTHOAuth consent requiredFrontend redirects user to OAuth provider

​
API Key Flow

When next_action is API_KEY, the user must submit their API key before the installation becomes active: 
curl -X POST https://api.orceum.com/v1/apps/{app_id}/installations/{installation_id}/api-key \
  -H "Authorization: Bearer USER_ORCEUM_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "api_key": "sk-user-provided-key"
  }'
Orceum will:
  1. Validate the key meets minimum requirements (≥8 chars, no spaces)
  2. Encrypt it with Fernet symmetric encryption
  3. Transition the installation to ACTIVE

​
Providing API Key Instructions

In your app’s config, you can guide users during this step: 
{
  "config": {
    "api_key_instructions": "Find your API key at Settings → API in your dashboard",
    "api_key_format_hint": "Starts with sk- and is 40 characters long"
  }
}

​
OAuth Flow

When next_action is OAUTH:
1

Orceum builds the authorization URL

Using your oauth_config.authorization_url, client_id, scopes, and state parameter.
2

User is redirected to consent screen

Orceum redirects to your OAuth provider’s authorization endpoint.
3

User grants access

The provider redirects to Orceum’s callback URL:
https://api.orceum.com/v1/apps/{app_id}/oauth/callback
4

Orceum exchanges the code

Makes the token exchange request using token_url, client_id, and client_secret.
5

Tokens stored and installation activated

Access and refresh tokens are encrypted and stored. Installation status → ACTIVE.
You must register https://api.orceum.com/v1/apps/{your_app_id}/oauth/callback as an allowed redirect URI with your OAuth provider, or the callback will be rejected.

​
Installation States

An installation transitions through these states:
StateDescription
PENDINGJust created — awaiting credential submission
ACTIVECredentials validated — app can execute actions
SUSPENDEDTemporarily disabled (admin action or repeated errors)
UNINSTALLEDUser removed the app — credentials cleared, tokens revoked
REAUTH_REQUIREDOAuth tokens expired/invalid — user must re-authenticate

​
State Transitions

PENDING
  ├─→ ACTIVE          (credentials submitted and validated)
  └─→ UNINSTALLED     (user cancels installation)

ACTIVE
  ├─→ SUSPENDED       (admin suspends or repeated failures)
  ├─→ UNINSTALLED     (user removes the app)
  └─→ REAUTH_REQUIRED (token refresh fails)

REAUTH_REQUIRED
  └─→ ACTIVE          (user re-authenticates)

SUSPENDED
  └─→ ACTIVE          (admin re-enables)

​
Re-authentication

If OAuth token refresh fails, the installation enters REAUTH_REQUIRED. Orceum generates a re-auth link the user can click to go through the OAuth flow again: 
curl -X POST https://api.orceum.com/v1/apps/{app_id}/installations/{installation_id}/reauth \
  -H "Authorization: Bearer USER_ORCEUM_TOKEN"

{
  "reauth_url": "https://api.orceum.com/v1/apps/{app_id}/oauth/start?state=..."
}

​
Lifecycle Webhooks

When a user installs or uninstalls your app, Orceum notifies your installation_webhook_url with a signed payload. See Lifecycle Webhooks for the full payload schema and verification guide.
